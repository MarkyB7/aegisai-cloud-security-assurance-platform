## 1. Purpose

This document defines the high-level system context for the AegisAI Cloud Security Assurance Platform.

Its purpose is to establish:

- who interacts with AegisAI
- which external systems exchange information with the platform
- where the AegisAI system boundary begins and ends
- the major information flows entering and leaving the platform
- the trust relationships that require further security analysis

This document intentionally avoids detailed AWS services and implementation components. Those details will be documented later in the logical, security, and deployment architecture views.

## 2. System Overview

AegisAI is an enterprise Cloud AI Security Assurance Platform designed to help organizations securely adopt, operate, and govern generative AI workloads.

The platform provides a centralized capability for:

- evaluating AI and cloud architectures against security requirements
- identifying prompt injection, data leakage, insecure retrieval, excessive permissions, and other AI-specific risks
- collecting security findings from cloud, application, identity, and AI environments
- supporting security investigation and automated response
- generating governance, risk, compliance, and executive assurance evidence
- integrating security controls into development and deployment workflows

At the system-context level, AegisAI is treated as a single platform. Its internal AWS services, application components, detection logic, data stores, and automation mechanisms are intentionally hidden from this view.

## 3. Human Actors

The following human actors interact with AegisAI or consume information produced by the platform.

### 3.1 Employee

Employees use AegisAI to submit approved security-related questions, request guidance, and access authorized AI-assisted capabilities.

### 3.2 AI Engineer

AI Engineers design, test, and deploy AI applications and retrieval-augmented generation workflows. They use AegisAI to evaluate prompts, models, data sources, permissions, and deployment configurations against security requirements.

### 3.3 Security Engineer

Security Engineers investigate findings, validate alerts, tune detections, review evidence, and manage approved remediation workflows.

### 3.4 Security Architect

Security Architects use AegisAI to assess proposed designs, evaluate trust boundaries, identify architectural risks, review security controls, and support architecture approval decisions.

### 3.5 Platform Administrator

Platform Administrators manage the AegisAI platform configuration, integrations, access controls, operational settings, and service availability.

### 3.6 Compliance Officer

Compliance Officers review control evidence, policy mappings, risk findings, audit records, and assurance reports generated by AegisAI.

### 3.7 Chief Information Security Officer

The Chief Information Security Officer consumes executive security posture, risk, compliance, and assurance reporting to support governance and risk-acceptance decisions.


## 4. External Systems

The following external systems exchange information with AegisAI. These systems exist outside the AegisAI platform boundary but are trusted to perform specific business or technical functions.

### 4.1 Identity Provider (IdP)

The Identity Provider authenticates users before they access AegisAI. Authentication is performed externally, while authorization decisions are enforced within the platform.

**Examples:** Microsoft Entra ID, Okta, AWS IAM Identity Center

### 4.2 AWS Cloud Environment

The AWS Cloud Environment hosts the AegisAI platform and provides the underlying cloud infrastructure on which the platform operates. At the system-context level, AWS is treated as the hosting environment rather than a collection of individual services.

### 4.3 Security Information and Event Management (SIEM)

The SIEM platform receives security events, alerts, audit records, and operational telemetry from AegisAI to support centralized monitoring, threat detection, investigation, and incident response.

**Examples:** Splunk, Microsoft Sentinel, Google SecOps, IBM QRadar

### 4.4 Source Code Platform

The Source Code Platform stores application code, Infrastructure as Code, security policies, and CI/CD workflows used to build and deploy AegisAI.

**Examples:** GitHub, GitLab, Azure DevOps

### 4.5 Infrastructure as Code (IaC) Platform

The Infrastructure as Code platform manages infrastructure provisioning and configuration through declarative code. It enables repeatable, auditable, and version-controlled deployments.

**Examples:** Terraform CLI, Terraform Enterprise


## 5. Platform Boundary

The AegisAI platform boundary defines the scope of systems, services, data, and security controls that are owned, managed, and protected by the AegisAI platform.

Components located inside the platform boundary are fully managed by the organization and are subject to the platform's security, governance, monitoring, and operational controls.

Systems located outside the platform boundary are considered external dependencies or actors. These entities exchange information with AegisAI but remain independently managed and require authenticated, authorized, and monitored interactions.

Establishing the platform boundary enables accurate threat modeling, trust boundary identification, security architecture design, and compliance scoping.


## 6. System Context Diagram Specification

The System Context Diagram will represent AegisAI as a single platform positioned at the center of the enterprise environment.

### 6.1 Central System

The diagram will contain one central system:

- **AegisAI Cloud Security Assurance Platform**

The platform will remain a single black box at this abstraction level. Internal AWS services, applications, data stores, security controls, and automation components will be documented in later logical and deployment architecture diagrams.

### 6.2 Human Actors

The following human actors will appear outside the AegisAI platform boundary:

- Employee
- AI Engineer
- Security Engineer
- Security Architect
- Platform Administrator
- Compliance Officer
- Chief Information Security Officer

The actor relationships will represent business requests, security reviews, operational actions, governance decisions, and reporting interactions.

### 6.3 External Systems

The following systems will appear outside the AegisAI platform boundary:

- Identity Provider
- Source Code Platform
- Infrastructure as Code Platform
- AWS Cloud Environment
- Security Information and Event Management platform

Each external system will have a direct, labeled relationship with AegisAI.

### 6.4 Platform Boundary

The platform boundary will surround only the AegisAI platform. Human actors and external systems will remain outside the boundary.

The boundary communicates that AegisAI owns, manages, secures, and monitors its internal capabilities while interacting with external actors and independently managed systems through controlled interfaces.

### 6.5 Relationship Labels

All diagram connections will use directional arrows and concise labels describing the information or capability exchanged.

Planned relationships include:

| Source | Destination | Relationship |
|---|---|---|
| Human Actors | AegisAI | Security requests, reviews, administration, approvals, and reporting |
| Identity Provider | AegisAI | Authenticated identity and claims |
| Source Code Platform | AegisAI | Application code, security policies, and CI/CD events |
| Infrastructure as Code Platform | AegisAI | Infrastructure plans, policy results, and deployment events |
| AWS Cloud Environment | AegisAI | Hosting capabilities, cloud findings, and operational telemetry |
| AegisAI | SIEM | Security findings, alerts, audit records, and telemetry |


## 7. Diagram Design Decisions

The System Context Diagram is intentionally organized into three logical layers to improve readability and communicate the platform's business relationships before implementation details.

### Layer 1 — Human Actors

Human actors are positioned above the platform because they initiate business interactions and consume platform capabilities.

Actors are grouped by organizational responsibility:

- Business Users
- Engineering
- Governance

### Layer 2 — AegisAI Platform

AegisAI is represented as a single platform to emphasize business capability rather than implementation details.

No AWS services or internal components are displayed at this level.

### Layer 3 — External Systems

External systems are positioned below the platform because they exchange information with AegisAI but do not represent business users.

Systems are grouped into:

- Identity
- Engineering
- Operations

This organization improves readability while emphasizing the relationships between people, external technology, and the AegisAI platform.

### Diagram Principle

Each architecture diagram must communicate one primary idea.

The System Context Diagram communicates **who interacts with AegisAI and which external systems exchange information with it**.

Implementation details are intentionally deferred to later architecture views.
